## 1. Executive summary

### Goal

Connect **empower.ankshaastra.com** to the **ankshaastra.com** operations hub so that:

| Data | Central location | Managed from |
|------|------------------|--------------|
| Customers | `customers` table | Admin → CRM |
| Orders / bookings | `orders` table | Admin → Orders |
| Payments | `payments` + `orders` | Admin → Orders / Webhooks |
| Invoices (GST) | `invoices` + `invoice_items` | Admin → Invoice Manager |
| Invoice PDFs | Supabase Storage `invoices` bucket | Auto-generated by hub API |
| GSTR-1 filing | Same invoices, filtered by month | Admin → GSTR Reports |
| Service subjects (baby name forms) | `service_persons` + `orders.metadata` | CRM (phase 2+) |

### Architecture principle: **One Supabase, many websites**

```
┌──────────────────────┐     ┌──────────────────────┐
│ empower.ankshaastra  │     │ ankshaastra.com      │
│ (checkout UI only)   │     │ (hub + admin panel)  │
└──────────┬───────────┘     └──────────┬───────────┘
           │  API (CORS)                 │
           └──────────────┬──────────────┘
                          ▼
              ┌───────────────────────┐
              │  Central Supabase     │
              │  · PostgreSQL (CRM)   │
              │  · Storage (PDFs)     │
              │  · Auth (admin users) │
              └───────────────────────┘
```

**Do not** run a second Supabase project for Empower production data. Empower becomes a **frontend spoke**; all writes go to the central project.

## 3. Central database schema (reference)

### Core tables for Empower integration

```
websites                    — registry (empower already seeded)
customers                   — CRM identity (email, phone, source_website)
orders                      — purchase record + metadata.formSnapshot
service_persons             — baby/subject details per order (CRM)
invoices                      — GST invoice header
invoice_items                 — line items (SAC 999799)
payments                      — Razorpay ledger
gst_config                    — single LLP config
gstr_export_runs              — filing audit log
email_logs / webhooks_log     — operations audit
```

### Key columns for multi-site

| Table | Column | Empower value |
|-------|--------|---------------|
| `orders` | `source_website` | `empower.ankshaastra.com` |
| `orders` | `metadata` | JSONB with `formSnapshot` (pincode, state, baby details) |
| `invoices` | `source_website` | `empower.ankshaastra.com` |
| `invoices` | `sac_code` | `999799` |
| `invoices` | `customer_state_code` | From checkout (required for GSTR) |
| `invoices` | `gstr_category` | Auto: B2B / B2CS / B2CL |
| `customers` | `source_website` | `empower.ankshaastra.com` |

### Order metadata contract (Empower checkout)

Store everything the baby-name form collects inside `metadata.formSnapshot`:

```json
{
  "fullName": "Parent Name",
  "email": "parent@example.com",
  "whatsapp": "+919876543210",
  "pincode": "110001",
  "customerState": "Delhi",
  "customerStateCode": "07",
  "dob": "2024-01-15",
  "tob": "10:30",
  "pob": "Delhi, Delhi, India",
  "sourceWebsite": "empower.ankshaastra.com"
}
```

Hub invoice engine reads this for place of supply, GSTR classification, and PDF billing block.

## 4. Storage architecture

### Current setup

| Item | Value |
|------|--------|
| Bucket | `invoices` (private) |
| Max file size | 10 MB per PDF (migration); config allows 50 MB |
| MIME | `application/pdf` only |
| Path pattern | `invoices/{safe_invoice_number}.pdf` |
| Access | Signed URLs (1 year TTL) stored in `invoices.pdf_url` |
| Upload | Hub `server/lib/invoice-storage.ts` via service role |

### Recommended path pattern (Phase 4)

Organize by site and year for easier bulk export and lifecycle policies:

```
invoices/
  empower.ankshaastra.com/
    2026/
      EYN26-27K-70001.pdf
  ankshaastra.com/
    2026/
      EYN26-27K-70002.pdf
```

Implementation: update `invoiceStoragePath()` in a later phase — **not required for Phase 1**.

### Optional future buckets

| Bucket | Purpose | When |
|--------|---------|------|
| `order-attachments` | Uploaded kundli PDFs, documents | Phase 3+ CRM |
| `report-deliverables` | Generated baby name reports | When AI/report delivery is centralized |
| `exports` | GSTR Excel temp files | Optional; currently generated in-memory |

## 5. Capacity & pricing analysis

### Supabase plan comparison (2026)

| Resource | **Free** | **Pro (~$25/mo)** | Notes |
|----------|----------|-------------------|--------|
| Database | 500 MB | 8 GB included | Shared CPU on free |
| File storage | **1 GB** | **100 GB** included | Invoice PDFs live here |
| Egress | 5 GB | 250 GB | PDF downloads count |
| Backups | None | 7-day daily | Production needs Pro |
| Pause risk | After 1 week idle | Never | **Risk for production** |
| Max projects (free org) | 2 active | Unlimited paid | Consolidate to **one** project |

Source: [supabase.com/pricing](https://supabase.com/pricing)

### Storage estimates (invoice PDFs)

Assume ~**70 KB** per invoice PDF (pdf-lib generated):

| Volume | PDF storage | DB growth (approx.) |
|--------|-------------|---------------------|
| 500 orders/year | ~35 MB | ~5–15 MB |
| 2,000 orders/year | ~140 MB | ~20–50 MB |
| 10,000 orders/year | ~700 MB | ~100–200 MB |
| 20,000 orders/year | ~1.4 GB | ~200–400 MB |

### Recommendation

| Stage | Plan | Why |
|-------|------|-----|
| **Development / pilot** | Free tier OK | Low volume; watch 1 GB storage cap |
| **Production (main + Empower + Miracle Baby)** | **Pro ($25/mo minimum)** | 100 GB storage, backups, no pause, headroom for 10k+ invoices/year |
| **Heavy report files later** | Pro + monitor usage | Add `order-attachments` bucket; archive old PDFs yearly |

### Cost optimization tips

1. **Single Supabase project** — avoid paying for Empower + Main separately.
2. **One invoice per order** — unique index already enforced.
3. **Private bucket + signed URLs** — no public CDN egress for PDFs unless needed.
4. **Annual archive** — move PDFs older than 7 years to cold storage (future).
5. **JSONB in `metadata`** — keep large blobs (images) in Storage, not DB rows.

## 9. Data flow diagrams

### New Empower purchase (Option A)

```
User on empower.ankshaastra.com
  → POST ankshaastra.com/api/create-order
       → INSERT orders (source_website=empower, status=pending)
  → Razorpay payment
  → POST ankshaastra.com/api/verify-payment
       → UPDATE orders (paid)
       → INSERT/UPDATE customers
       → invoice-engine: INSERT invoices + PDF upload Storage
       → email PDF to customer
  → Admin sees order + invoice (filter: empower)
  → GSTR Reports includes invoice in monthly export
```

### GSTR-1 (all sites, one filing)

```
All invoices (ankshaastra + empower + miraclebaby)
  → filtered by invoice_date (month)
  → normalized (SAC 999799, state codes)
  → validated (errors only block export)
  → Excel: B2B, B2CS aggregate, B2CL, SAC Summary
  → filed under 09AAFFE7583B1ZD
```
